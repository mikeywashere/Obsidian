## Key Architecture Decisions

### Why Blazor WASM is NOT Orchestrated

- Obsidian.Web is pure Blazor WebAssembly — runs entirely in browser
- No server-side host component to orchestrate
- For local dev: `dotnet run` Obsidian.Web separately; it calls API via configured base URL
- Trinity configured WASM to use environment variables for API discovery

### Connection String Configuration

- SQLite connection string configured as parameter resource (no native Aspire integration)
- AppHost sets connection string for Obsidian.Api to use
- Removed `IsAspireHost` property to avoid deprecated workload warning
